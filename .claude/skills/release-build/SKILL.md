# release-build

`scripts/release-build.sh {android|ios|web} <appdir> [--install]`

## Android

`cd <appdir>/android && ./gradlew assembleRelease` with the repo's debug-keystore params (see
script). Artifact: `<appdir>/android/app/build/outputs/apk/release/app-release.apk`. The release
APK embeds the JS bundle from `.build/native/commonjs` and runs WITHOUT Metro; `--install` does
`adb install -r` + launches. Ensure the Fable native build is fresh first (fable-rebuild-verify).

## Release-only failure modes (engineering log, session 10). Check these when release breaks and debug works:

1. LibClient-only deps not resolvable in release bundle -> map them in the app's
   `metro.config.js` `extraNodeModules` (e.g. @react-native-picker/picker, async-storage).
2. Reanimated/worklets live only in LibClient/node_modules -> add as direct app deps
   (+ @babel/core + babel.config.js).
3. Local images crash RCTImageView ("Value for uri cannot be cast from Double to String") ->
   numeric require() assets must pass as bare RN source, not {uri}.
4. `ReferenceError: Property 'crypto' doesn't exist` -> add react-native-get-random-values,
   import it FIRST in index.js.

## iOS

A Release IPA for own-device install is produced in three deterministic stages. The
`ios-preflight.sh` script gates stages 1-2 and MUST pass before archiving.

**Signing prerequisite (one-time per clone).** `DEVELOPMENT_TEAM` is never committed; it lives in
git-ignored `ios/Local.xcconfig` (`DEVELOPMENT_TEAM = <team id>`), pulled in by the committed
`ios/Config.xcconfig` project-level base config. New app without `ios/` wiring yet:
`Meta/LibScaffolding/scripts/wire-ios-signing.pl <app>/ios/<App>.xcodeproj/project.pbxproj`. Full
detail in `runbooks/ios.md` (Signing setup). Deployment target is iOS 18.0.

### Stage 1: preflight (deterministic gate)

```
.claude/skills/release-build/scripts/ios-preflight.sh <appdir>
```

Checks, each printed PASS/FAIL/WARN with the exact value tested; any FAIL exits 1 with an
actionable fix and you must not archive until it is green:

1. `.xcworkspace` exists, `Pods/` installed
2. scheme resolves (prefers the scheme matching the app/workspace stem; `xcodebuild -list` prints
   schemes alphabetically so the first is often a pod like `AppCenter`)
3. `PRODUCT_BUNDLE_IDENTIFIER` (from `project.pbxproj`), `ios/Config.xcconfig` present, and
   `DEVELOPMENT_TEAM` (from git-ignored `ios/Local.xcconfig`; WARNs if hardcoded in the pbxproj)
4. an `Apple Development` codesigning identity exists in the keychain (with its team id)
5. `DEVELOPMENT_TEAM` is logged into Xcode -> Settings -> Accounts (else `-allowProvisioningUpdates`
   / a UI archive cannot fetch a profile; fails with "No Account for Team" / "No profiles for X")
6. a provisioning profile for `<TEAM>.<BUNDLE_ID>` is cached (else the first archive must generate
   it); the target device UDID is listed from `ProvisionedDevices`
7. the Podfile `post_install` disables pod-target signing (see iOS failure modes below)
8. the Fable native bundle is fresh: `.build/native/commonjs/Bootstrap.js` mtime >= newest `src/*.fs`
9. ADVISORY: `index.js` gates `configSourceOverrides.native.js` on `__DEV__` (else Release ships
   dev config; does not block)
10. ADVISORY: reports `CFBundleDisplayName` so the home-screen label is not mistaken for a wrong-app
    reference (e.g. AppTodo's label is "Todo", not "AppTodo")

### Stage 2: archive

The first archive (or any archive after a signing change) MUST be done in Xcode UI, not `xcodebuild`
CLI: a free personal team cannot have its signing certificate generated by `xcodebuild
-allowProvisioningUpdates` (profiles yes, cert no), and the CLI `codesign` of embedded frameworks
hits `errSecInternalComponent` because it cannot unlock the login keychain non-interactively. The
Xcode UI session holds the keychain unlock and generates the cert on first archive.

In Xcode: open `<appdir>/ios/<App>.xcworkspace`, set the preflight's scheme, set the destination to
**Any iOS Device (arm64)**, then **Product -> Archive**. The `.xcarchive` lands under
`~/Library/Developer/Xcode/Archives/<date>/`.

### Stage 3: export to IPA (wrap, do not exportArchive)

`xcodebuild -exportArchive` re-signs embedded frameworks and hits `errSecInternalComponent` from the
CLI for the same keychain-unlock reason as stage 2. A Development IPA is just a signed `.app` in a
`Payload/` zip, so wrap the already-signed app from the archive directly:

```
.claude/skills/release-build/scripts/release-build.sh ios <appdir>
```

The script picks the newest `.xcarchive` for the app, verifies it is signed + has an embedded
`main.jsbundle` + `embedded.mobileprovision`, and writes `<bundleid>.ipa` to `dist/ios/`. No
re-signing, no App Store / "release testing" gate (that gate is TestFlight/App Store and needs a
$99 paid program; a Development IPA for your own provisioned device does not).

Install: drag the `.ipa` into Xcode -> Window -> Devices and Simulators -> your iPhone, or
`xcrun devicectl device install app --device <UDID> <ipa>`, then on the phone trust the team under
Settings -> General -> VPN & Device Management.

### iOS release-only failure modes

- **Pod targets fail "No signing certificate 'iOS Development' found"** for every RN pod: pods
  inherit the legacy `iOS Development` identity that no modern cert matches. Fix: add to the
  Podfile `post_install` a loop that sets `CODE_SIGNING_ALLOWED=NO`,
  `CODE_SIGNING_REQUIRED=NO`, `CODE_SIGN_IDENTITY=''` on every pod target's build configurations,
  then `pod install`. The app target re-signs embedded frameworks at the `[CP] Embed Pods
  Frameworks` step, so pods never need their own signing. (Already applied to AppTodo + Gallery.)
- **"No Account for Team XXXXX"** / **"No profiles for Y were found"**: `DEVELOPMENT_TEAM` in the
  pbxproj is not logged into Xcode Accounts, or the bundle id is registered to a different team. Add
  the Apple ID that owns the team in Xcode -> Settings -> Accounts.
- **`errSecInternalComponent`** during `codesign` of a framework in `xcodebuild` (archive or
  `-exportArchive`): the CLI cannot unlock the login keychain non-interactively. Archive via the
  Xcode UI; export via the wrap (stage 3), not `-exportArchive`.
- **`__DEV__`-gated prod config missing** (advisory): a Release build has `__DEV__` false; if
  `index.js` loads `configSourceOverrides.native.js` unconditionally the app ships dev config
  (e.g. `BackendUrl=http://localhost:5000`). Mirror the gallery's
  `if (__DEV__) { ... } else { require("./configSourceOverrides.native.prod.js") }`.
- **Stale `main.jsbundle`**: the "Bundle React Native code and images" phase bundles from
  `.build/native/commonjs`; if the Fable native build is older than `src/*.fs`, the archive ships
  old JS. The preflight (check 8) catches this; fix with `(cd <appdir> && ../../eggshell build-native)`.

## Web

`eggshell package-web` from the app dir is the eggshell web release command: it runs `build-lib`
for the web target, runs Fable in `package` mode (plain `webpack`, NOT `webpack-dev-server`; this
is the production bundle path, mirrored from `Meta/LibFablePlus/src/index.ts`), then packages the
`dist-template` into a standalone dist (server.js, package.json, run.sh) via
`Meta/LibRtCompilerFileSystemBindings/src/packageApp.ts`. Run it as
`(cd <appdir> && node <repo-root>/eggshell package-web)` (no dedicated CLI flag takes an appdir;
the CLI resolves the project from cwd). Smoke check: serve the output statically, page boots, no
dev-server references, no console errors.
