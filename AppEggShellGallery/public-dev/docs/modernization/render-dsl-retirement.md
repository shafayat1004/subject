# Render DSL Retirement (Goals A, C, D)

Goal A is retiring the `.render` template language entirely. Goals C (component verbosity) and D
(directory structure standardization) ride alongside it and are addressed here. The accessibility and
automation bar that every converted component must meet is also summarized here; see
[Accessibility](./accessibility/index.md) for the full plan.

---

## What the render DSL is

Historically, components were authored as `.render` files: an XML-ish template language (`rt-if`,
`rt-let`, `rt-map`, `rt-mapi`, `rt-match`, `rt-class`, `{expr}` interpolation, `{=fsharp}` escapes,
`<dom.div>` and `<Component>` tags). These were never JavaScript; they always compiled to F#
(`*.Render.fs` under `_autogenerated_/`). Later, the team began writing render functions directly in
F#, and the DSL is now being retired.

The `.render` trio for a component looked like:

```
Components/Button/Button.typext.fs     # Props record + component class + Actions
Components/Button/Button.styles.fs     # styles (F# CSS-in-JS DSL, class cascade)
_autogenerated_/.../Button.Render.fs   # generated render function
```

The modern target is a single `Foo.fs` using `[<Component>]` static members, with no `.typext.fs`,
no `.styles.fs`, and no autogenerated files. See the [component conventions](./fsharp/component.md)
and `CLAUDE.md` for the full recipe.

### DSL machinery deletion: not yet started

The remaining closure step is deleting the DSL build infrastructure:

- Strip the render compile from the build (`processAllRender`, watch hooks in
  `LibRtCompilerFileSystemBindings`).
- Delete `Meta/AppRenderDslCompiler`, `SuiteDsl/LibDsl`, `Meta/LibRenderDSL`.
- Remove the `renderdsl` CLI command and the `render` section of `eggshell.json`.

This is deferred until the full gallery audit pass (web + Android) confirms no regressions. Deleting
this stage also removes the ~548-subprocess render-compiler stage from every build, which is a
significant build-speed win (see [Build Performance](./modernization/build-performance.md)).

## Goal A closure: what "done" looks like

A component is done when:

- Single `Foo.fs` with `[<Component>]`; no `.render` / `.typext.fs` / `.styles.fs`.
- `Themes.Set<LC.Foo.Theme>` in `DefaultComponentsTheme.fs`; no parallel `FooStyles.Theme.Customize`.
- All repo callers migrated (framework, LibRouter, gallery).
- Interactive surfaces use `LC.Pressable` with `label` (and `testId` if automation-relevant).
- Gallery content page pure F# if the component is showcased.
- No style leaks, no new console warnings.
- `dotnet build` Web Debug green for affected projects.
- an [Engineering Log](./knowledge-base/engineering-log.md) entry if a new gotcha was discovered.

## Goal C: component verbosity

The `[<Component>]` + hooks path already points the way. After the DSL is gone, lean into it:

- Scaffolding templates for the common component shapes (leaf display, interactive control, layout
  container, stateful).
- Helpers for the subscribe-render pattern (`LC.With.Subjects`, `AsyncData`).
- Fewer mandatory `Estate` / `Pstate` / `Actions` declarations for trivial components.

Currently: approximately 16 components still call the `TapCapture` shim directly rather than
`LC.Pressable` with explicit `label` / `role` / `testId`. Once these are converted, `TapCapture.fs`
can be deleted.

---

## Goal D: directory structure standardization

The structure is already fairly consistent after Goal A. The key conventions:

```
Components/<Name>/<Name>.fs          # single file, [<Component>] static members
ComponentRegistration.fs             # auto-generated by eggshell build-lib
LibClient.fsproj                     # compile order: source files before autogenerated
```

After DSL retirement, the `_autogenerated_/` tree disappears entirely, which is the single biggest
structural simplification. The remaining conventions will be codified in a written guide and in the
corrected scaffolder (Goal B, see [Scaffolding](./modernization/scaffolding.md)).

## The component conversion playbook (summary)

Full skeletons: see the converted components (`Tabs.fs`, `Heading.fs`) and [F# Components](./fsharp/component.md). The quick decision tree:

1. **Preflight:** grep for external `FooStyles` consumers and `Foo.Make` / `LC.Foo` callers. If
   external consumers exist, plan a cluster migration rather than a solo conversion.
2. **Classify:** interactive (9.2), input/form (9.7), layout/container (9.3), stateful (9.6),
   pseudo-stateful (9.5), or leaf display (9.1). Modifiers: responsive (9.4), native-divergent (9.9),
   dialog (9.8), ThirdParty (9.10).
3. **Build** from the archetype skeleton. Port `.styles.fs` to `module private Styles` with
   `makeViewStyles` / `makeTextStyles` / `*.Memoize`. Meet the no-style-leaks and no-console-warnings
   quality bar.
4. **A11y/automation:** satisfy the acceptance bar for the archetype (see below).
5. **Delete + register:** remove `.render`, `.typext.fs`, `.styles.fs`, the whole
   `_autogenerated_/Components/Foo/` dir, and the `RegisterRender` / `RegisterStyles` lines; fix
   `.fsproj` compile order; run `eggshell build-lib`.
6. **Validate:** `dotnet build LibClient/src/LibClient.fsproj -c "Web Debug"` must be 0 errors. For
   app-facing components, also run a full gallery `eggshell dev-web`.
7. **Mirror:** update the gallery page to pure F# (Rule 10 in `CLAUDE.md`); append an [Engineering Log](./knowledge-base/engineering-log.md)
   entry if a new gotcha appeared.

**Hard "do not" list:**

- Never copy `_autogenerated_/.../Foo.Render.fs` into `Foo.fs`. That preserves
  `findApplicableStyles` / `__componentStyles` soup and is not a modernization.
- Never `open ReactXP.LegacyStyles` in a modern file (breaks `makeViewStyles` CE with FS0041 yield
  errors).
- Never declare public `State` / union types at module top with `[<AutoOpen>]` (leaks union cases
  globally; collides with `DefaultComponentsTheme`). Nest under `module LC = module Foo`.
- Never leave a `FooStyles.Theme.Customize` shim alongside a new `Themes.Set`.
- Never patch `.render` or hand-edit `_autogenerated_/.../*.Render.fs` for feature work; convert to
  pure F# instead.

## Accessibility and automation bar

Every converted component must satisfy the row for its archetype:

| Archetype | Label | Role | State | testId | Live region | Keyboard/focus (v2) |
|-----------|-------|------|-------|--------|-------------|---------------------|
| Leaf display | only if status-bearing | - | - | optional | if it announces | - |
| Interactive control | required | required | disabled/selected as applicable | required | - | focusable, Enter/Space |
| Input/form leaf | required (associated) | field role | disabled/checked/selected | required | error -> assertive | tab order |
| Layout/container | - | - | - | optional (scroll regions) | - | scroll containment |
| Dialog/overlay | title -> label | `Dialog` | - | required | open -> polite | Escape (have); trap (v2) |

Cross-cutting requirements for all interactive components:

- Single labeled press target via `LC.Pressable` (no visual-plus-invisible-overlay split).
- Stable `testId` survives refactors; slugged ids via `A11ySlug.testId prefix label`.
- Press/nav actions recorded to `UiActionLog`; component registers in the interactive registry via
  `componentName`.
- `window.__eggshell.<App>.uiSnapshot()` lists the component (testId/label/role/state) when mounted.

For the full accessibility plan, including the `[safe]` / `[rnw-blocked]` / `[web-only]` migration
tagging and the post-RNW targets, see [Accessibility](./accessibility/index.md).

## Style-leak rule (hard rule, codified)

**Never let a `makeViewStyles` / `makeTextStyles` run inside render.** No inline `styles = [| makeViewStyles {...} |]`, none inside a component body, `Pointer.State`, or `With.ScreenSize` callback.

- Static styles: top-level `let`.
- Parametrized styles: `ViewStyles.Memoize` / `TextStyles.Memoize`, **keyed on primitives** (Color /
  int / bool / small DU), never on whole `Theme` / `Colors` records (fresh record refs defeat the
  cache).
- A memoized lambda parameter must not share a name with a CE operation (`height`, `color`,
  `fontSize`, `top`, `left`, `bottom`) or it will not compile; rename it (`itemHeight`, `labelColor`,
  etc.).

Full rationale and recipe: `docs/fsharp/styling.md` ("Avoiding style leaks") and the [Engineering Log](./knowledge-base/engineering-log.md)
entry 2026-06-28.

## React key warnings rule

When passing a static array of sibling elements as a component's children, route it through
`castAsElementAckingKeysWarning` / `tellReactArrayKeysAreOkay` (both call
`React.Children.toArray` to inject stable keys) instead of a bare `[| ... |]` / `castAsElement`.

Long-term fix: have the `element` / `elements` CE builder auto-key multi-child output structurally
(Goal C, Goal A). Until then, convert offenders when touching them. Do not mass-suppress the warning
without real keys.
