react-native-web with the New Architecture" effort (branch `modernization/rnw`). This page is the
single place that tracks, in detail, **what is done** and **what remains**. For the narrative and
gotchas see the [Engineering Log](./knowledge-base/engineering-log.md) (sessions 8 and 9); for the
one-line dashboard see [Current Status](./modernization/index.md).

**Target stack (shipped):** React **19.2.3**, react-native **0.86.0**, react-native-web **0.21.2**,
RNGH **3.0.2**, Reanimated **4.5.1** + worklets **0.10.2** + Moti **0.30.0**, New Architecture
(Fabric / TurboModules / Bridgeless) **on**. Toolchain host Fable 5.4.0 / .NET 10 SDK.

## Remaining work (detailed)

Ordered by value/dependency. Each item: **what / files / steps / verify / risk**.

### RW1 — Gallery native build (Android + iOS) [large ThirdParty workstream]

The gallery's RN-0.86 config is applied and its **web build is verified**, but the **native** build
is blocked by unmaintained third-party native modules.

- **First blocker (Android):** `assembleDebug` fails at
  `LibPushNotification/Client/node_modules/react-native-push-notification/android/build.gradle:5` --
  `Could not find method jcenter()` (Gradle 9 removed `jcenter()`). `react-native-push-notification@8.1.1`
  is the unmaintained latest (2022, AGP 3.2.0) -> needs a **patch-package patch or replacement**
  (Notifee, or `@react-native-community/push-notification-ios` + FCM), not a bump.
- **The long tail:** the gallery autolinks **~20 ThirdParty native modules** (`Map`,
  `ReactNativeVisionCamera`, `BackgroundGeolocation`, `ImagePicker`, `FilePicker`, `ReactNativeContacts`,
  `ReactNativeDeviceInfo`, `ReactNativeCodePush`, `SunmiPrint`, `FacebookPixel`, `ReCaptcha`,
  `GoogleAnalytics`, `QRCode`, ...), several unmaintained -- each a likely Gradle-9 / RN-0.86 / Fabric-C++
  blocker.
- **Files:** `ThirdParty/*` package versions + patches; `LibClient/package.json` /
  `AppEggShellGallery/package.json`; per-module `android/`; for iOS, `AppEggShellGallery/ios/Podfile` +
  `pod install`.
- **Steps:** triage each of the ~20 modules -> bump / patch-package / replace / drop; rebuild
  `assembleDebug` and iterate; then iOS `pod install` + build.
- **Verify:** gallery `assembleDebug` green + launches on device (`fabric:true`); iOS builds on the
  simulator.
- **Risk:** high, iterative, multi-day; several modules need replacement, not just bumps. The gallery is
  web-primary, so this is lower urgency than it looks.

### RW2 — Adopt Reanimated in components / field defect 5 [risky rewrite of a working shared control]

- **Context:** `LibClient/src/Components/SegmentedControl/SegmentedControl.fs` **already** slides its
  thumb via the framework RN-Animated primitive (`animatedTranslateX` + `Animation.Timing` 220ms on
  selection; `AnimatedValue.SetValue` during drag; `thumbStatic` under `LC.With.ReducedMotion`). So
  "adopt Reanimated" means replacing working RN-Animated code in a control used by **every app on both
  platforms** with new Moti/Reanimated F# bindings -- high blast radius, needs per-platform (web +
  native) animation verification, and Moti-on-web depends on the reanimated-web path.
- **Field defect 5** ("dark-mode toggle jumps instead of slides") was reported on the **old RN 0.76**
  stack. A plausible cause is a theme-change **remount** resetting `thumbInitializedRef` -> `syncThumb`
  jump (a small logic fix), NOT something Reanimated is required for.
- **Recommendation:** (1) re-test defect 5 on the RN-0.86 release build; (2) if it persists, fix the
  remount/init logic (small); (3) treat a full Moti/Reanimated migration of shared components as a
  separate, device-verified follow-up. The Reanimated 4 infra is installed and native-verified.
- **Files:** `SegmentedControl.fs`, `SuiteTodo/AppTodo/src/Components/Route/Todos.fs` (`TodoSwipeShell`);
  a framework-owned JS worklet shim only if UI-thread math is needed (`ThirdParty` / `TypesJs.fs`; never
  author worklets in F#).

### RW3 — PerformancePlayground: modernize or retire [decision]

`SuitePerformancePlayground/AppPerformancePlayground/android` is **pre-AndroidX legacy** (Gradle 6.3,
build-tools 28, support-lib 28) and cannot build RN 0.86 as-is. Options: (a) retire the app; (b)
regenerate its `android/` (and `ios/`) from the RN-0.86 template + bump `package.json`; (c) keep it
web-only. **Needs a product decision** (is PP still used?). Left untouched for now.

### RW4 — Scaffolding `eggshell create-app` (Goal B) [gates end-to-end scaffold verify]

Templates are updated to RN 0.86 (see Done), but `eggshell create-app` is **separately broken** (see
[Scaffolding](./modernization/scaffolding.md) -- the "drift table"). Fixing Goal B is what lets a
generated app build web + native end-to-end and adds the `create-app -> dev-web` CI smoke gate.

### RW5 — React 19 / DOM validation follow-ups [minor]

- **Nested `<h1>`**: React 19 validates DOM nesting more strictly and warns about `<h1>` inside `<h1>`
  from the Heading role emission (a latent semantic issue, now visible). Fix `Heading` to not nest.
- **React "unique key" warnings** (CLAUDE.md rule I): route static child arrays through
  `castAsElementAckingKeysWarning`.

### RW6 — Ship [after the above]

Broader consumer check for any other `App*`/`Suite*` that build native, then open the PR from
`modernization/rnw` to `main` with the stack-move summary and breaking notes.
