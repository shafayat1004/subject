# Roadmap

Future work for EggShell. More of a prioritized "future work" list than a committed schedule. Items
carry a **status** from a 2026-06 repo check (**In progress** / **Partial** / **Not done** / **Not
decided**); completed work is omitted.

The repo currently runs **Fable 4** and **.NET 7**, and the gallery app has no `.render` files left.

## Active initiative (current Fable 4 / .NET 7)

These ship together as the current phase. They reinforce each other and are intentionally scoped to the
current toolchain (no .NET 10 / Fable 5 yet).

### A. Retire the render DSL

Convert framework `.render` components to pure F#, then delete the DSL machinery entirely.

* Upgrade `eggshell convert-component` to write the `.fs` in place and delete the `.render` (today it
  only prints to stdout).
* Convert the framework-owned `.render` first (`LibClient`, `LibUi*`, `LibRouter`, `ThirdParty`),
  reviewing the generated F# for readability.
* Once converted: strip the render compile from the build, delete `Meta/AppRenderDslCompiler`,
  `SuiteDsl/LibDsl`, and `Meta/LibRenderDSL`, and simplify `eggshell.json` / `initialize`. This also
  removes the per-file render-compiler subprocess stage (~548 spawns repo-wide), a build-speed win (see E).

  **Partial:** the gallery is 100% pure F#; LibClient is ~95% converted. **9** `.render` files remain
  (Input.Text, the Picker cluster, Dialog.Base, ContextMenu.Dialog). Bulk auto-convert is blocked on
  styling/theming, not syntax.

### B. Fix scaffolding

`eggshell create-app` must produce a working, modern app — currently it does not. A correct template is,
by definition, the canonical modern EggShell app, so this also anchors C and D.

* Rewrite the `eggshell.json` / `App.fsproj` / `package.json` templates to the current schema + web
  dev-server tooling.
* Stop emitting `.render` from `create-component/route/dialog`; emit pure F#.
* Decouple from company-specific libraries; ship a generic minimal app.
* Auto-update the `.fsproj` on `create-component` (today it is a manual, error-prone step).
* Add a `create-app → dev-web` CI smoke test so scaffolding cannot silently rot again.

  **In progress.**

### C. Reduce component verbosity

Lean into the `[<Component>]` + hooks path. After the DSL is gone: scaffolding templates for the common
component shapes, helpers for the subscribe-render pattern, and fewer mandatory `Estate`/`Pstate`/
`Actions` declarations for trivial components.

  **In progress.**

### D. Standardize frontend directory structure

Structure is already fairly consistent. Removing `_autogenerated_` (post-DSL) is itself a big
simplification. Codify the remaining conventions in a written guide and in the corrected scaffolder (B).

  **In progress.**

### E. Speed up the frontend build

Frontend build cost is a compound of independent stages. Near-term, independent levers:

* Fingerprint-cache the LibStandard precompile (currently reruns even when nothing changed) — likely the
  biggest warm-build win.
* Re-enable parallel type-check (`--noParallelTypeCheck` is currently set); debounce the webpack trigger
  instead.
* Parallelize the dependency-tree build (currently serial).
* Lighten dev source maps (`cheap-module-source-map`).

The structural wins come from A (deleting the render stage) and F (Fable 5 caching/codegen).

  **Not started / partial.**

## Platform & backend (later)

Explicitly later than the active initiative. F is the baseline that unblocks the rest.

### F. Platform baseline: .NET 10 + Fable 5

.NET 7 is **out of support** (EOL May 2024), so this is also a security/maintenance item. Bump
`global.json` and all `<TargetFramework>` to net10, move to Fable 5, and resolve the package upgrades
(Orleans, Giraffe, SqlClient, Fable.React, etc.). Fable 5 targets net10, uses direct MSBuild cracking,
adds `Pojo` bindings, and improves caching/codegen. Risk is concentrated in the custom
`Meta/FablePlugins` (the `[<Component>]` plugin) — verify it compiles against Fable 5's plugin API.

  **Not done.**

### G. PostgreSQL + Orleans 3.7 → 7 upgrade

One coordinated backend storage workstream. Orleans 3.7→7 is a **hard, non-rolling** migration (wire
protocol + grain identity changed) — stand up a new cluster and decommission the old one. The Postgres
storage rewrite touches the same membership/clustering/storage seams, so do both **once, together**,
against the target Orleans version. Order: decide .NET-LTS + Orleans target → abstract the connection
factory + provider selection → port DDL/stored-procs + membership tables → Postgres storage handlers →
run the full simulation suite in `TestDataSeeding` mode against Postgres.

  **Not done.**

### H. ReactXP decision

ReactXP is archived upstream; `@chaldal/reactxp` is the only living fork. This gates the frontend
modernization budget.

* **Option A — invest in the fork:** no disruption to hundreds of components, full control of cadence,
  but you are the sole maintainer and must backfill modern features by hand. A legitimate "buy time"
  holding pattern.
* **Option B — migrate the primitive layer to React Native for Web:** rejoins a maintained ecosystem
  (animation, gestures), but is a refactor, not a drop-in. The contained seam is the wrapped `RX.*`
  primitives in `LibClient/src/ReactXP`; if those keep their F# signatures, most component code is
  untouched.

Either way, a real **animation story** (spring/physics motion, gesture-driven and layout/route
transitions) is the highest-visibility frontend win.

  **Not decided.**

### Security hardening

A focused framework auth/SQL/crypto pass; verify before acting (defaults often have a subtle intended path).

* **Verify first:** possible **default-allow** when a subject/view has no session handling configured
  (`maybeSessionHandling = None` → `Grant`); and schema/lifecycle/ecosystem **names interpolated** into
  SQL via `sprintf` (identifiers are not parameterized — add a whitelist/`QUOTENAME`-style escape).
* **Follow-ups:** CORS wildcard-subdomains used with `AllowCredentials`; 2-year cookie expiry; harden
  the AES-GCM XML encryptor before production wiring.

  **Not started.**

* switch the `eggshell` toolchain to dotnet

  **Not done:** `LibClient` still references `LibLifeCycleTypes` and Subject-oriented components.

## Higher Level Abstractions

* develop the Executor pattern to its logical conclusion. This is fairly important, it's what allows
  us to fully automate in progress and error states.

  **Partial:** Executors are central to `LC.Form.Base`, buttons, and async flows
  ([Executors](../how-to/executors.md)). Many call sites still manage `InProgress` manually.

* Finish implementing LibAutoUi. Since last we looked at it, we've learned a lot about building
  input forms, have developed a handful of component to support basic types, so now just one
  more major iteration on LibAutoUi should make it usable (estimated 1~2 weeks of work)

  **Not done:** `LibAutoUi/` exists but no app `.fsproj` references it. Use `LC.Form.Base` + typed
  inputs in production apps.

## Tooling

* PowerCycle is broken, it doesn't work for library components, only current app components

* RenderDsl VSCode extension is broken, it doesn't work for namespaced components

* error mapping for RenderDsl. As part of this, we'll need to replace the standard .net XML parser
  that we are currently using, because it does not provide us with line number information.

  **Not done:** Errors still surface in `_autogenerated_/.../*.Render.fs` (see [Tools](../tools/index.md)).
  (Moot once A retires the DSL.)

* make snippets for all input types

  **Partial:** `Meta/renderdsl.code-snippets` covers several inputs but not every `LC.Input.*` type.

* `eggshell rename-prop` would be really nice. Unfortunately it's not possible to implement this
  using XSLT, since the .render files are meant for human consumption, and thus cannot be randomly
  reformatted by the XSLT processor, which happens with quotes, whitespace, CDATA sections and more.
  So this will have to wait until we deal with error mapping, at which point we'll have to build an
  XML processor that is aware of the underlying text, and using that to implement prop rename and
  similar features.

  **Not done:** Not in `eggshell` CLI.

* allow to have < and > and & inside XML attributes — this will go against the XML spec, but will
  make it much easier to work with types, particularly as used in `rt-prop`, and boolean expressions.

* fsproj component registration autogeneration
    * auto-generating a standalone components file, and requiring that from fsproj
      would be the ideal way to do this. Can also automate ordering, but that would
      require parsing of all components at once and running an algo on the
      dependency graph.

  **Partial:** `ComponentRegistration.fs` is auto-generated per lib (`eggshell build-lib`).
  `.fsproj` compile order is still manual.

* enforce single quotes for RenderDSL attributes (and either exclude `rt-type-parameters`, or
  allow to escape single quotes somehow)

* consider a possible design for type-safe class names
